# 需求文档：云相册「照片 × 地点」足迹与回忆

| 文档属性 | 说明 |
|----------|------|
| 产品 | CloudPhoto 云相册 |
| 状态 | 草案（供评审与排期） |
| 版本 | v0.1 |
| 范围 | 客户端（Android / iOS）+ 与服务端契约说明 |

---

## 1. 背景与问题陈述

用户在长期使用云相册的过程中，积累了大量照片。仅凭时间线或文件夹浏览，**难以按「在哪里拍的」这一维度唤起记忆**。将**用户已上传至个人云相册的照片**与**可解释的拍摄地点**结合，可以在地图、时间线、统计视图中帮助用户**回忆过往经历**（旅行、聚会、日常路线等）。

**数据来源约束（已确认）**：足迹与地点展示所依赖的照片列表与元数据，来自**用户通过 App 上传到自己的云相册后的服务端列表**，不以「仅扫描本机相册」为唯一数据源；换机登录后行为应一致。

---

## 2. 产品愿景与目标

### 2.1 愿景

让用户用「地点」作为一把钥匙，打开云相册里的记忆：**一张图、一个点、一段故事**。

### 2.2 目标（可验收）

1. **可发现**：用户能在一个明确入口（如「足迹 / 地图」）看到带有位置信息的照片在地图上的分布。
2. **可浏览**：从地图或列表进入某一地点（或聚合区域），能查看该处的照片集合，并跳转到单张详情。
3. **可回忆**：结合**拍摄时间**（或上传时间作为降级）与地点，支持按时间、按城市等维度的简单统计或浏览，辅助「想起那天在那儿」。
4. **可控**：用户能理解位置数据从哪来、存哪、能否关闭；关闭后符合隐私承诺的行为可验证。

### 2.3 非目标（本期不做或另立项）

- 社交地图、公开分享他人轨迹。
- 实时导航、路线规划、POI 深度运营（可作为后续增强）。
- 无用户授权下的第三方位置画像或广告定向。

---

## 3. 用户画像与典型场景

| 角色 | 诉求 |
|------|------|
| 旅行爱好者 | 按城市/国家看自己历年照片分布，点开回忆行程。 |
| 普通用户 | 偶尔发现「原来这张是在那儿拍的」，增强相册情感价值。 |
| 隐私敏感用户 | 一键关闭位置参与足迹；历史数据按策略处理（见第 8 节）。 |

**典型用户故事（示例）**

- 作为已登录用户，我希望在地图上看到**我云相册里带位置的照片**，以便快速想起去过的地方。
- 作为用户，我希望点击地图上的某一区域后看到**该处的缩略图列表**，以便浏览回忆。
- 作为用户，我希望在设置里**关闭基于拍摄地点的足迹功能**，以便不展示、不利用位置信息（在能力范围内）。

## 4. 功能需求

### 4.1 数据与列表（核心）

- **列表数据源**：与现有云相册照片列表一致（分页、筛选、相册维度若已有则继承）。
- **位置字段**：每条照片记录可选包含：
  - **经纬度**（WGS84，与常见 EXIF 一致；若后端统一为 GCJ02，需在契约中写明并在地图上使用一致坐标系）。
  - **是否有位置**：布尔或「经纬度非空」推导。
  - **拍摄时间** `takenAt`（优先用于回忆时间轴）；若无则用 `createdAt`（上传/入库时间）作为展示与排序降级，并在 UI 上弱提示「时间为上传时间」（可选）。
- **无位置照片**：在足迹模块中**不强制显示在地图上**；可提供「未标注地点的照片」入口（可选，二期）。

### 4.2 上传与写入（若当前仅列表有字段、上传未写入，则需补齐）

- **上传路径**：在用户选择上传的文件时，客户端**在授权前提下**读取 EXIF 中的 GPS 与拍摄时间（若存在），随上传流程提交给服务端，或由服务端在接收原图后解析（二选一，需定案）。
- **失败与降级**：无 EXIF、无 GPS、解析失败时，服务端存空；不影响上传成功。
- **去重与更新**：替换照片、编辑元数据（若产品提供）时，位置字段与服务端存储一致更新。

### 4.3 地图足迹（MVP）

- **地图展示**：在地图上以标记点或聚类展示「有位置」的照片；缩放级别变化时聚类合并/展开（性能目标见第 9 节）。
- **交互**：点击标记 → 预览若干张缩略图或进入「该点照片列表」；支持进入单张详情（沿用现有照片详情页能力）。
- **平台**：Android 与 iOS 分别使用平台推荐地图能力（或团队已选统一 SDK），**地图 SDK 不进入 KMP shared**，共享逻辑仅处理数据模型与聚合。

### 4.4 回忆向浏览（MVP+）

至少满足以下之一（排期时拆分为 P0 / P1）：

- **按时间 + 地点**：例如「2024 年 5 月」下列出出现过的城市/区域，点击进入照片网格。
- **按城市/行政区聚合**：展示去过的地方列表及每处照片数量（需**逆地理编码**时：可由服务端返回 `displayName`，或客户端在展示层调用地理编码服务，注意配额与隐私）。

### 4.5 设置与开关

- **总开关**：例如「在足迹/地图中使用照片拍摄地点」关闭后：
  - 地图足迹入口可隐藏或进入后展示空态 + 说明；
  - **不再将新的 GPS 元数据上传**（若当前上传逻辑会带上位置）；
  - 已存储在云端的位置是否删除或仅隐藏，由**隐私策略**定案（见第 8.3）。
- **系统权限说明**：若读取本地 EXIF 需相册/文件权限，文案与系统弹窗时机符合各应用商店审核要求。

## 5. 与现有工程的数据契约说明（对齐 CloudPhoto）

当前仓库中 `PhotoDTO` 已包含 `latitude`、`longitude`、`takenAt`、`createdAt` 等字段；领域模型 `Photo` 若尚未承载经纬度与 `takenAt`，实现阶段需统一：**列表接口 → DTO → 领域模型 → UI**，避免地图层直接依赖 DTO。

建议在接口契约中明确：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| latitude / longitude | number，可空 | 否 | 无 GPS 时为 null |
| takenAt | epoch ms，可空 | 否 | 无 EXIF 拍摄时间时为 null |
| createdAt | epoch ms | 是 | 上传/服务端时间，作降级展示 |

（若增加 `locationPrecision`、`countryCode`、`cityName` 等，可减轻客户端逆地理压力，列为增强项。）

## 6. 客户端架构原则（KMP）

1. **shared**：照片位置相关字段、列表筛选（例如「仅有位置」）、按城市/按日聚合的**纯计算逻辑**、统一错误类型（`ApiResult` 等）。
2. **expect/actual**（如需要）：仅用于「上传前从本地文件读 EXIF」等平台能力，不在 Composable / SwiftUI 中散落实现。
3. **composeApp / iosApp**：地图渲染、权限请求、逆地理（若放客户端）、空态与引导。

---

## 7. 用户体验与界面要点

- **空态**：无带位置照片时，说明「仅当照片含拍摄地点信息时显示」；可引导打开开关或检查相机是否保存位置（简短，避免冗长教程）。
- **加载**：列表分页加载；地图聚类避免一次性拉全量（见第 9 节）。
- **无障碍**：地图标记与列表项具备可读标签（内容描述 / accessibilityLabel）。
- **设计系统**：与项目现有主题色、间距、圆角一致（参见 `.cursor/rules/12-ui-design-system.mdc`）。

---

## 8. 隐私、安全与合规

### 8.1 告知同意

- 首次使用足迹或首次上传带位置照片时，展示**简明隐私说明**：用途（回忆、地图展示）、存储位置（用户个人云相册）、不与第三方共享（除非另有独立同意）。
- 应用商店隐私标签与隐私政策正文需同步更新。

### 8.2 数据最小化与安全

- 传输：**HTTPS**；不在 URL 查询串中传递精确经纬度。
- 日志：禁止打印完整经纬度、原始 EXIF 块；如需排错使用 photoId 与脱敏坐标。
- 存储：服务端按现有用户数据隔离与鉴权；仅本人可拉取自己的照片位置列表。

### 8.3 用户关闭与删除（需产品定案）

下列策略选其一并在文档定稿时写明：

- **A**：关闭后仅客户端不展示，云端保留（再次开启可恢复）。
- **B**：关闭后触发服务端清除该用户照片的 `latitude/longitude`（或匿名化），不可恢复。
- **C**：提供独立「清除所有足迹位置数据」操作。

---

## 9. 性能与可用性

- **列表**：地图所需数据应支持**分页**或**按地理范围查询**（若后端提供 bbox 接口，优先），避免单次加载万级点。
- **聚类**：客户端聚类在点数过大时需限流或后端聚合（阈值与方案在技术评审定）。
- **弱网**：地图瓦片与列表请求失败时，错误提示可重试；不崩溃。

## 10. 指标与验收（建议）

| 类型 | 示例指标 |
|------|-----------|
| 采用 | 足迹入口 DAU、从地图进入详情的转化率 |
| 质量 | 带位置照片占比（运营侧）、上传写库成功率 |
| 隐私 | 关闭开关使用率、相关客诉量 |

**验收检查清单（节选）**

- [ ] 仅登录用户可见自己的足迹数据。
- [ ] 列表与地图坐标一致、无系统性偏移（坐标系约定验证）。
- [ ] 关闭位置开关后的行为与隐私文案一致。
- [ ] 双端（Android / iOS）核心路径一致：有数据 / 无数据 / 错误 / 关闭开关。

## 11. 分期建议（路线图）

| 阶段 | 内容 |
|------|------|
| **P0** | 列表展示位置字段；地图 MVP（点或聚类 + 进入列表/详情）；上传写入 GPS/takenAt（若尚未贯通）。 |
| **P1** | 按月份/城市回忆浏览；空态与引导优化；性能优化（范围查询或后端聚合）。 |

## 12. 依赖与开放问题（评审清单）

1. **坐标系**：后端与客户端、地图底图是否统一为 WGS84 / GCJ02？
2. **拍摄时间来源**：仅 EXIF，还是允许用户手动修改？
3. **逆地理**：服务端返回文案 vs 客户端请求地图厂商 API？成本与合规？
4. **历史数据**：是否提供「批量补解析」任务（用户重新触发同步）？
5. **国际化**：城市名称多语言与边界情况（争议地区展示策略）。
